# Supabase Schema Migration Guide

## Overview
This guide walks you through migrating your Supabase database from the **OLD schema (bigint IDs)** to the **NEW schema (UUID IDs)**.

**Status**: The old schema is currently on Supabase. We're migrating to the new cleaner schema.

---

## 🔄 Migration Steps

### Step 1: Access Supabase SQL Editor
1. Go to **https://supabase.com/dashboard/project/vylmcqmxpxqkldosowrs**
2. In the left sidebar, click **SQL Editor**
3. Click **New Query**

### Step 2: Run the Migration Script
1. Copy the entire contents of `database/MIGRATION_to_new_schema.sql`
2. Paste it into the SQL Editor
3. Click **Run** (or Ctrl+Enter)
4. Wait for completion (should take a few seconds)

### Step 3: Verify Migration Success
After the script completes, run this verification query in a new SQL query:

```sql
SELECT 'Roles' as table_name, COUNT(*) as count FROM role
UNION ALL SELECT 'Users', COUNT(*) FROM "user"
UNION ALL SELECT 'Customers', COUNT(*) FROM customer
UNION ALL SELECT 'Categories', COUNT(*) FROM category
UNION ALL SELECT 'Products', COUNT(*) FROM product;
```

**Expected Results:**
- Roles: 3 (admin, cashier, inventory_staff)
- Users: 1 (admin)
- Customers: 0
- Categories: 7
- Products: 0

## ⚠️ Troubleshooting

### Q: "Table already exists" error
**A**: The old tables weren't properly dropped. Open a new query and run:
```sql
DROP TABLE IF EXISTS notification CASCADE;
-- ... (repeat for all old tables)
```

### Q: Foreign key constraint errors
**A**: The tables have interdependencies. Run the migration script again - it handles the order automatically.

### Q: UUID vs bigint errors in backend
**A**: Make sure your backend is using the new table names (singular) and talking to the new schema. Restart the backend after migration.
